# MMORPG Board

Django-приложение для сообщества MMORPG, где пользователи могут регистрироваться, 
подтверждать email, создавать объявления, оставлять отклики и управлять ими. 
Проект реализован как классическое server-rendered web-приложение на Django с авторизацией через сессии,
CSRF-защитой, ORM, фоновыми задачами и Docker-окружением.

Также настройки вынесены в переменные окружения, 
добавлена поддержка PostgreSQL/Redis, Docker Compose, CI-проверки, тесты, защита прав доступа и оптимизация
запросов к базе данных.

## Возможности

- регистрация пользователей и обязательная email-верификация через `django-allauth`;
- авторизация через Django sessions;
- пользовательский профиль с аватаром и дополнительными данными;
- создание, редактирование, публикация и удаление объявлений;
- категории объявлений;
- черновики, доступные только владельцу;
- поиск, фильтрация и пагинация объявлений;
- отклики пользователей на объявления;
- запрет отклика на собственное объявление;
- принятие или отклонение откликов владельцем объявления;
- email-уведомления;
- фоновые задачи через Celery и Redis;
- еженедельная рассылка через Celery Beat;
- защита от пользовательского HTML через очистку контента;
- Django Admin для управления данными;
- Docker-окружение для локального запуска с PostgreSQL и Redis;
- CI-проверки качества кода и тестов через GitHub Actions.

## Основные backend-задачи

В проекте реализована серверная логика для полноценной доски объявлений:

- CRUD для объявлений;
- разграничение прав доступа между владельцем объявления и другими пользователями;
- проверка прав при изменении статуса отклика;
- фильтрация и поиск через Django ORM;
- пагинация списков;
- оптимизация запросов через `select_related` и `prefetch_related`;
- миграции базы данных;
- фоновые задачи для email-уведомлений;
- безопасная работа с пользовательским контентом.

## Безопасность

В проекте используются стандартные механизмы безопасности Django:

- CSRF-защита для POST-запросов;
- session authentication;
- обязательная email-верификация;
- проверка владельца объекта перед редактированием и удалением;
- запрет на изменение чужих объявлений и откликов;
- хранение секретов через `.env`;
- `.env` исключен из Git;
- добавлен `.env.example` без реальных секретов;
- пользовательский HTML очищается перед сохранением.

JWT в проект не добавлялся осознанно, потому что это не SPA и не mobile API. Для server-rendered Django-приложения session authentication и CSRF подходят лучше.

## Оптимизация базы данных

В проекте были устранены потенциальные проблемы N+1-запросов.

Для связанных объектов используются:

- `select_related` для связей ForeignKey;
- `prefetch_related` для обратных и many-to-many связей;
- агрегаты для подсчета откликов;
- фильтрация на уровне ORM вместо обработки данных в Python.

Это уменьшает количество SQL-запросов при отображении списков объявлений и связанных данных.
